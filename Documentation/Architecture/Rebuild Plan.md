# Rebuild Plan

A from-scratch rebuild of the Delightful Design System with the architecture described in [[Core Concepts & Fundamental Technologies]]. The existing repo (`delightful-design-system`) becomes read-only reference. A new repo inherits every design decision, color value, and component design — but none of the structural debt.

---

## Premise

The existing system works visually but is structurally inverted. The showcase HTML is the source of truth, and everything else is extracted from it via regex. The rebuild inverts this: data and source files are authoritative, and the showcase is assembled from them.

**What carries over:** Every design decision. All 174 OKLCH color values (across 44 primitives and ~55 semantic mappings, with ~77 additional component/foundation tokens bringing total CSS custom properties to ~176). All 47 component designs. All 59 animation keyframes. The neo-brutalist aesthetic. The 3-tier token model. The interaction patterns. The research. Everything that makes Delightful *Delightful*.

**What doesn't carry over:** The monolithic HTML file as source of truth. The regex sync pipeline. Manual port creation. Backfilled tests. The extraction problem.

**Reference material:**
- `delightful-design-system.html` — component designs, token values, interaction patterns
- `delightful-color.html` — palette explorer functionality
- `delightful-motion.html` — animation catalog
- `delightful-animation.html` — physics demos
- `palettes/` values in existing ports — hex authority for terminal themes
- Research docs — rationale for every technical decision

### Key structural decisions

**`src/tokens.css` is generated, everything else in `src/` is authored.** The CSS emitter writes tokens.css from palette JSON. Components, foundation, motion, and animation files are hand-written source files that reference tokens via `var()`. This is the one-directional dependency: data → tokens → components → documentation.

**One component per CSS file.** Each file in `src/components/` defines exactly one component. No cross-component dependencies beyond shared tokens. `index.css` re-exports everything for bundle consumers.

**Ports are generated outputs.** The `ports/` directory contains only emitter outputs (plus authored content for the Claude plugin's skills/agents). Changing a color means editing `palettes/delightful.json` and running `npm run build`. No manual hex translation ever.

**Documentation is assembled.** The `docs/` HTML files import source CSS/JS and render interactive documentation. They're the last thing built, not the first.

**Packages are thin wrappers.** Each `packages/` directory has a `package.json` and a build script that copies/bundles from `src/`. No duplication — packages reference the same source files.

## Build Pipeline

```
npm run build
  │
  ├── 1. Validate palette JSON against schema (ajv)
  ├── 2. Run WCAG contrast checks on semantic pairs
  ├── 3. Emit tokens.css (OKLCH custom properties, Tier 1 + 2, light/dark)
  ├── 4. Emit platform ports:
  │       ├── VS Code theme JSON (hex via culori)
  │       ├── Obsidian theme CSS (OKLCH, --d-* namespace)
  │       ├── Ghostty config (hex, from terminal authority values)
  │       ├── iTerm2 .itermcolors (RGB floats)
  │       ├── Starship TOML (hex)
  │       └── Tailwind preset (var() references)
  ├── 5. Bundle components:
  │       ├── dist/components.css (all components concatenated)
  │       └── dist/components/*.css (individual files)
  └── 6. Assemble documentation pages (inject CSS/JS into HTML templates)

npm run test
  │
  ├── Palette validation (schema + contrast)
  ├── Emitter output correctness (generated files match expectations)
  ├── Component rendering (per-component visual + interaction tests)
  ├── Motion accessibility (prefers-reduced-motion compliance)
  ├── Port consistency (terminal palettes match cross-platform)
  └── Documentation assembly (pages render, no console errors)
```

### Phase 1: Color Data (Day 1, afternoon – Day 2)

**Goal:** The palette exists as validated JSON. This is the single source of truth for all color.

**Deliverables:**

**1.1 — Palette JSON** (`palettes/delightful.json`)
- Extract all 174 color values from the existing `delightful-design-system.html`
- Structure:
  ```json
  {
    "$schema": "./palette.schema.json",
    "$schemaVersion": 1,
    "name": "Delightful",
    "version": "1.0.0",
    "author": "Kyle Snavely",
    "primitives": {
      "neutral": { "0": "oklch(1.00 0.000 0)", "25": "oklch(0.988 0.006 70)", ... },
      "pink": { "100": "oklch(0.920 0.060 350)", ... },
      "red": { ... },
      "gold": { ... },
      "cyan": { ... },
      "green": { ... },
      "purple": { ... }
    },
    "semantic": {
      "light": {
        "bg-page": "oklch(0.982 0.008 70)",
        "text-primary": "oklch(0.200 0.015 60)",
        "accent-primary": "oklch(0.640 0.270 350)",
        ...
      },
      "dark": { ... }
    },
    "terminal": {
      "light": {
        "foreground": "#1b150f",
        "background": "#fdf8f3",
        "cursor": "#f600a3",
        "ansi": ["#1b150f", "#ed324b", "#22a448", "#febf00", "#00a6c0", "#f600a3", "#17c0d6", "#e8ddd3", "#6b5e52", "#ff6b7f", "#44d66a", "#ffe066", "#33cfea", "#ff33c0", "#55e8f0", "#fdf8f3"]
      },
      "dark": { ... }
    }
  }
  ```
- OKLCH authoritative for `primitives` and `semantic`
- Hex authoritative for `terminal` (hand-tuned values from existing ports)

**1.2 — JSON Schema** (`palettes/palette.schema.json`)
- JSON Schema draft 2020-12
- Validates structure, required fields, OKLCH string format, hex format for terminal
- Specific validation rules:
  - OKLCH format regex: `^oklch\(\d+\.\d+\s+\d+\.\d+\s+\d+(\.\d+)?\)$`
  - Required primitive families: neutral, pink, red, gold, cyan, green, purple
  - Minimum stops per accent family: 5 (100, 200, 300, 400, 500)
  - Neutral family minimum: 14 stops (0–950)
  - Required semantic categories: backgrounds, text, borders, accents, status, shadows, focus/overlay
  - Terminal hex format regex: `^#[0-9a-fA-F]{6}$`
  - Required terminal fields: foreground, background, cursor, ansi (16-element array)

**1.3 — Validation script** (`emitters/validate.mjs`)
- Schema validation via ajv
- OKLCH parse-ability (every OKLCH string is well-formed)
- WCAG contrast checks for specific pairs (both light and dark modes):
  - `text-primary` on `bg-page` ≥ 4.5:1
  - `text-primary` on `bg-surface` ≥ 4.5:1
  - `text-secondary` on `bg-page` ≥ 4.5:1
  - `text-on-accent` on `accent-primary` ≥ 4.5:1
  - `text-on-accent` on `accent-danger` ≥ 4.5:1
  - `text-on-accent` on `accent-success` ≥ 4.5:1
  - `text-on-gold` on `accent-gold` ≥ 4.5:1
  - `text-on-accent` on `accent-cyan` ≥ 4.5:1
  - `text-on-accent` on `accent-purple` ≥ 4.5:1
  - `border-default` on `bg-page` ≥ 3:1 (non-text contrast)
- Terminal distinctness: adjacent ANSI colors are visually distinguishable

**1.4 — Shared utilities**
- `emitters/palette-loader.mjs` — reads and validates palette JSON
- `emitters/oklch-utils.mjs` — OKLCH parsing, hex conversion (wraps culori), gamut clamping

**1.5 — Tests** (`tests/palette.spec.ts`)
- Schema validation passes
- All OKLCH strings parse correctly
- WCAG contrast thresholds met
- Terminal colors are distinct

**Acceptance:** `npm run build` validates the palette. `npm test` verifies all constraints. The palette JSON contains every color value from the existing system — nothing lost.

### Phase 2: Emitter Pipeline (Days 2–3)

**Goal:** One command generates all platform outputs from palette JSON.

**Deliverables:**

**2.1 — CSS emitter** (`emitters/css.mjs`)
- Reads palette JSON
- Writes `src/tokens.css`:
  - `@layer primitives` with all primitive custom properties
  - `@layer semantic` with `:root` (light) and `[data-theme="dark"]` selectors
  - Shadow tokens (structure is CSS — only color values come from palette)
- Output is valid CSS with no build step needed by consumers

**2.2 — VS Code emitter** (`emitters/vscode.mjs`)
- Reads palette JSON
- Converts OKLCH → hex via culori with `clampChroma()` for sRGB gamut
- Writes `ports/vscode/themes/delightful-light-color-theme.json` and `delightful-dark-color-theme.json`
- ~324 color property mappings (editor, UI, syntax, terminal ANSI, debug)
- Reference: existing `vscode-theme/scripts/generate-themes.mjs` (port the mappings, not the structure)

**2.3 — Obsidian emitter** (`emitters/obsidian.mjs`)
- Reads palette JSON
- Writes `ports/obsidian/theme.css`
- Renames `--primitive-*` → `--d-*` (Obsidian namespace convention)
- Includes Style Settings plugin configuration block
- OKLCH values used directly (no hex conversion)

**2.4 — Terminal emitters** (`emitters/ghostty.mjs`, `emitters/iterm2.mjs`)
- Read palette JSON (specifically `terminal.light` and `terminal.dark` hex values)
- Ghostty: writes plain-text config with `palette = index=hex` format
- iTerm2: writes Apple plist XML with RGB float values (hex → float conversion)
- Both must produce identical ANSI palettes (enforced by test)

**2.5 — Starship emitter** (`emitters/starship.mjs`)
- Reads palette JSON terminal hex values
- Writes `ports/starship/starship.toml` with color-coded prompt segments

**2.6 — Tailwind emitter** (`emitters/tailwind.mjs`)
- Reads palette JSON
- Writes Tailwind preset with `var()` references to token custom properties
- 1:1 token coverage check: every semantic token has a Tailwind utility

**2.7 — Orchestrator** (`emitters/orchestrator.mjs`)
- Runs validation first (fail-fast)
- Runs all emitters in sequence
- Reports: files written, tokens emitted, any warnings
- Wired to `npm run build`

**Shared emitter contract:** Every emitter implements the same interface:
```js
emit(palette, options) → { files: [{ path, content }], warnings: [] }
```
Each emitter receives the parsed palette object and returns an array of files to write plus any warnings. The orchestrator handles file I/O and warning aggregation — emitters are pure functions that produce output strings.

**2.8 — Tests** (`tests/tokens.spec.ts`, `tests/ports.spec.ts`)
- `tokens.spec.ts`: generated `tokens.css` contains all expected custom properties with correct values
- `ports.spec.ts`: Ghostty and iTerm2 ANSI palettes are identical; VS Code theme has all required keys; Obsidian uses `--d-*` namespace

**Acceptance:** `npm run build` generates all platform outputs from a single `palettes/delightful.json`. Changing a color value in the JSON and rebuilding updates every port automatically. Zero manual hex translation.

### Phase 6: Animation System (Day 6–7)

**Goal:** Spring physics, FLIP, and particle systems as importable JS modules.

**Deliverables:**

**6.1 — Spring physics** (`src/animation/spring.js`)
- `springStep(current, target, velocity, { stiffness, damping, mass })` — core physics function
- `createSpring(config)` — returns an animator with `.start()`, `.stop()`, `.update()`
- Three presets: snappy, standard, gentle (matching existing demo configs)
- Respects `prefers-reduced-motion` (instant-completes when enabled)

**6.2 — FLIP utilities** (`src/animation/flip.js`)
- `measureFirst(element)` → snapshot of position/dimensions
- `measureLast(element)` → snapshot after layout change
- `invert(first, last)` → calculate transform to fake old position
- `play(element, invertTransform, config)` → animate from inverted to final

**6.3 — Particle system** (`src/animation/particles.js`)
- `createParticleSystem(canvas, config)` — initializes canvas rendering
- `splashEffect(element, event)` — the button/card click splash from existing system
- Configurable: particle count, velocity range, colors (reads from CSS custom properties)

**6.4 — Module index** (`src/animation/index.js`)
- Re-exports all animation modules
- Tree-shakeable: consumers import only what they use

**6.5 — Tests**
- Spring physics: unit tests for convergence, energy conservation, reduced-motion bypass
- FLIP: layout transition produces correct transform values
- Particles: canvas rendering doesn't throw, respects reduced-motion

**Acceptance:** Each module works independently. `import { createSpring } from './animation/spring.js'` works without importing the full system. All physics match existing demo behavior.

### Phase 8: Packages (Day 8–9)

**Goal:** Installable npm packages under `@delightful` scope.

**Prerequisite:** Verify `@delightful` npm scope is available. Do this on Day 1. If unavailable, choose alternative before writing any `package.json`.

**Deliverables:**

**8.1 — `@delightful/tokens`** (`packages/tokens/`)
- `dist/tokens.css` — copy of `src/tokens.css` (all Tier 1 + 2 custom properties)
- `dist/tokens.json` — W3C Design Token Community Group (DTCG) format, generated from palette JSON. Uses the standard `{ "$type": "color", "$value": "oklch(...)" }` structure with nested groups for families and tiers.
- `dist/tokens.mjs` — ES module export of token values as JS object
- `dist/tokens.d.ts` — TypeScript declarations
- Zero dependencies

**8.2 — `@delightful/tailwind`** (`packages/tailwind/`)
- `dist/preset.js` — copy of Tailwind emitter output
- Peer dependency: `tailwindcss`, `@delightful/tokens`

**8.3 — `@delightful/css`** (`packages/css/`)
- `dist/components.css` — bundled (all components)
- `dist/components/*.css` — individual component files
- `dist/foundation.css` — typography, spacing, radius, shadows
- `dist/motion.css` — all 59 animations
- `dist/reset.css` — base reset
- Peer dependency: `@delightful/tokens`

**8.4 — `@delightful/react`** (deferred to later phase)
- Thin typed wrappers around CSS components
- `forwardRef`, variant props, compound patterns
- Peer dependency: `@delightful/tokens`, `@delightful/css`, `react`
- Start with Button + Card to establish patterns, then expand

**8.5 — Build scripts**
- `npm run build:packages` — copies/bundles from `src/` into each `packages/*/dist/`
- `npm run publish:packages` — publishes all packages at same version (lockstep)

**8.6 — Package exports maps**

Each package's `package.json` must specify conditional `exports` fields so bundlers and consumers resolve the correct entry point:

- **`@delightful/tokens`**: `"exports": { ".": { "import": "./dist/tokens.mjs", "require": "./dist/tokens.cjs", "types": "./dist/tokens.d.ts" }, "./css": "./dist/tokens.css", "./json": "./dist/tokens.json" }`
- **`@delightful/css`**: `"exports": { ".": "./dist/components.css", "./components/*": "./dist/components/*.css", "./foundation": "./dist/foundation.css", "./motion": "./dist/motion.css", "./reset": "./dist/reset.css" }`
- **`@delightful/tailwind`**: `"exports": { ".": "./dist/preset.js" }`

**8.7 — Tests**
- Each package's `dist/` output exists and is well-formed
- tokens.json matches tokens.css values
- CSS bundle contains all component imports
- Package.json has correct peer dependencies

**Acceptance:** `npm install @delightful/tokens @delightful/css` and a single CSS import gives a consumer the full design system. Individual component CSS files work in isolation.

## What This Eliminates

| Problem from v1 | Why it existed | Why it can't exist here |
|-----------------|---------------|------------------------|
| Regex sync script with 2 bugs | HTML was source, tokens extracted via regex | JSON is source, CSS is generated. No regex parsing. |
| 8,006-line monolith | Everything defined in one file | Components are individual CSS files |
| "CSS extraction" as highest-risk task | Components trapped in HTML | Components were never in HTML |
| Manual hex translation for ports | No emitter pipeline | Every port generated by an emitter |
| Zero-assertion visual tests | Tests backfilled after components | Tests written alongside each component |
| Toggle token violation | No mechanical enforcement | Linter catches primitive references in component layer. CI blocks merge. |
| No CI/CD | Added as afterthought | Created on Day 1, runs on every push |
| 5 sources of color truth | Colors copied to each port manually | One source: `palettes/delightful.json` |
| v0.7.0 migration (12 issues) | Retrofitting pipeline onto existing code | Pipeline is the starting point. No migration. |
| v0.8.0 extraction (10 issues) | Extracting from monolith for packaging | Source files are the packages. Nothing to extract. |
