## 1. Purpose

`@delightful/tokens` is the distributable npm package containing all design tokens for the Delightful Design System. It is the foundational package that every other `@delightful/*` package depends on. It has zero dependencies.

---

## 2. Inputs

| Source | Path | Description |
|--------|------|-------------|
| CSS tokens | `src/tokens.css` | Generated by the CSS emitter from `palettes/delightful.json`. Contains all Tier 1 (primitive) and Tier 2 (semantic) custom properties with light/dark theme support. |
| Palette JSON | `palettes/delightful.json` | The color authority file. Used to generate the JSON and JS outputs. |

---

## 3. Outputs

All outputs are written to `packages/tokens/dist/`.

| File | Format | Description |
|------|--------|-------------|
| `tokens.css` | CSS | Copy of `src/tokens.css`. All custom properties in `@layer primitives` and `@layer semantic`. |
| `tokens.json` | JSON | W3C Design Token Community Group (DTCG) format. Nested groups for token families and tiers. Each token uses the `{ "$type": "color", "$value": "oklch(...)" }` structure. |
| `tokens.mjs` | ES Module | ES module export of all token values as a JavaScript object. Mirrors the JSON structure. |
| `tokens.d.ts` | TypeScript | TypeScript declarations for `tokens.mjs`. Every token name is a typed key -- no `any` types. |

---

## 4. Key Decisions (to be resolved)

- **W3C DTCG format depth**: How deeply do we nest the JSON? Flat list of tokens with dot-notation names, or nested groups matching the primitive/semantic/component tier structure?
- **TypeScript declaration depth**: Do we type individual token values as string literals (`"oklch(0.982 0.008 70)"`) or as `string`? Literal types give autocomplete but increase bundle size.
- **CJS support**: Do we need a `tokens.cjs` for `require()` support? The exports map currently includes a `require` condition pointing to `tokens.cjs`. If CJS is dropped, this simplifies the build.
- **Minimum Node version**: What Node.js version do we declare in `engines`? Node 18+ is the likely minimum (LTS as of the build date).

---

## 5. Open Questions

- Do consumers need the raw OKLCH strings, or should `tokens.mjs` also provide hex-converted values for environments that don't support OKLCH?
- Should `tokens.json` include non-color tokens (spacing, radius, motion timing) or only color tokens?
- Is the `./json` export path the right name, or should it be `./dtcg` to signal the format?

---

## 6. Build Script

The `build:packages` script performs these steps for the tokens package:

1. Copy `src/tokens.css` to `packages/tokens/dist/tokens.css`.
2. Read `palettes/delightful.json` and generate `tokens.json` in W3C DTCG format.
3. Generate `tokens.mjs` from the JSON structure (export default the token object).
4. Generate `tokens.d.ts` from the JSON structure (type every key).
5. Optionally generate `tokens.cjs` (CommonJS wrapper).

---

## 7. Exports Map

```json
{
  "name": "@delightful/tokens",
  "exports": {
    ".": {
      "import": "./dist/tokens.mjs",
      "require": "./dist/tokens.cjs",
      "types": "./dist/tokens.d.ts"
    },
    "./css": "./dist/tokens.css",
    "./json": "./dist/tokens.json"
  }
}
```
