01 — Philosophy

Bauhaus clarity, digital warmth

Kyle DS draws from the Bauhaus principle that form follows function, but adds a distinctly digital warmth. Primary colors are used with the confidence of Mondrian, the precision of a Swiss grid, and the playfulness of a well-crafted tool that respects your intelligence.

Blue for action
Blue is the workhorse. It signals primary actions, active states, focus rings, and links. When a user needs to know what to do next, blue is the answer.
Red for emphasis
Red is reserved for destructive actions and critical alerts. It demands attention without crying wolf. Used sparingly, it carries weight every time.
Yellow for delight
Yellow highlights, celebrates, and draws the eye to moments of joy. Achievements, new features, and warm callouts. It's the smile in the system.
Do
Use one primary color per component. Let blue lead for actions. Add red or yellow only when their specific meaning applies. White space is a feature.
Don't
Combine all three primaries in a single component. Use red for non-destructive actions. Apply yellow to large surfaces. Let color substitute for clear hierarchy.

Usage Guidelines
Do
Blue for primary buttons, links, focus rings, and active nav states. Red only for delete, disconnect, or error states. Yellow for badges, highlights, and celebratory moments. Green for success confirmation.
Don't
Red for cancel buttons (use ghost instead). Yellow as a background for large areas. Multiple accent colors competing in the same component. Raw primitive tokens in component code (use semantic tokens).

Charts that tell stories.

Visualization Guidelines

Do
  • Use sequential single-hue ramps for ordered data (revenue, time)
  • Keep categorical palettes to 8 or fewer series
  • Use blue as the default/first series in every chart
  • Provide text labels for key data points — don't rely on color alone
  • Use the diverging scale (blue↔red) for positive/negative comparisons
  • Include axis labels, legends, and units on every chart
  • Use opacity (0.08–1.0) for heatmaps instead of discrete color steps
  • Embed sparklines in tables for scannable trend data
Don't
  • Use red and green as the only distinction (colorblind-unsafe)
  • Apply 3D effects, gradients, or decorative chart elements
  • Use pie charts with more than 5 slices (switch to horizontal bars)
  • Mix categorical and sequential palettes in the same view
  • Start bar chart Y-axes at non-zero values without clearly marking it
  • Use yellow for large chart areas — it's a highlight, not a fill
  • Animate chart data on load in BI dashboards (respect scan patterns)
  • Use stroke-only chart elements below 2px — they disappear in dark mode
